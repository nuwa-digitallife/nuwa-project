# 个人知识库系统 - 开发规划

## 项目目标

构建一个自动化的个人知识管理系统：
- 爬取微信文章等内容
- 自动分类存储
- 支持交互式问答
- 最终同步到 Notion

## 用户需求

### 1. Notion 环境
- ✅ 已有 Notion 账号
- ✅ Mac 上运行 Notion App
- ⏳ 待配置：Integration Token 和 Database

### 2. 分类体系
主题分类（7个核心主题）：
- 人工智能
- 商业
- 金融
- 个人成长
- 历史
- 哲学
- 文学

**分类策略**：
- 自动分类（基于文章内容关键词/标题分析）
- 用户可手动纠正分类
- 支持多标签（一篇文章可属于多个主题）

### 3. 问答记录格式
- 每篇文章对应一个独立的问答文件
- 记录完整的对话历史
- 包含：时间戳、问题、回答、文章引用

### 4. 实现路线
**阶段 1**：本地知识库（当前优先）
**阶段 2**：Notion 集成（稳定后）

### 1.5 核心功能模块

#### 模块 1：文章爬取与存储
- 输入：文章 URL
- 处理：
  1. 调用 `connect_chrome.py` 爬取
  2. 提取元数据（标题、作者、来源）
  3. 自动分类
  4. 生成唯一 ID
  5. 保存为 Markdown
  6. 更新索引
- 输出：文章路径、分类结果

#### 模块 2：自动分类器
- 输入：文章标题 + 正文前 500 字
- 处理：
  1. 关键词匹配打分
  2. 可选：使用 LLM 辅助分类（Claude API）
  3. 计算最高分类别
- 输出：主分类 + 标签列表

#### 模块 3：问答交互
- 输入：用户问题 + 文章 ID
- 处理：
  1. 读取文章内容
  2. 基于文章内容回答问题
  3. 追加到对应的 qa 文件
- 输出：答案 + 更新问答记录

#### 模块 4：知识库管理
- 重新分类文章
- 搜索文章（按标题/内容/标签）
- 查看统计信息
- 导出为其他格式

## 阶段 2：Notion 集成（未来）

### 2.1 Notion 结构设计

**Database: 个人知识库**

| 字段 | 类型 | 说明 |
|------|------|------|
| 标题 | Title | 文章标题 |
| 分类 | Select | 7个主题之一 |
| 标签 | Multi-select | 多个标签 |
| 来源 | URL | 原文链接 |
| 作者 | Text | 作者名 |
| 爬取日期 | Date | 采集时间 |
| 阅读状态 | Select | 未读/在读/已读 |
| 正文 | Page | 子页面存储正文 |
| 问答记录 | Relation | 关联到问答 Database |

**Database: 问答记录**

| 字段 | 类型 | 说明 |
|------|------|------|
| 问题 | Title | 问题内容 |
| 回答 | Text | 回答内容 |
| 文章 | Relation | 关联到知识库 |
| 时间 | Date | 提问时间 |
| 笔记 | Text | 用户补充笔记 |

### 2.2 同步策略
- **方式 1**：实时同步（爬取后立即推送到 Notion）
- **方式 2**：批量同步（本地编辑后统一同步）
- **方式 3**：双向同步（Notion 修改也能同步回本地）

### 2.3 所需工具
- `notion-client` Python 库
- Notion Integration Token（需要用户创建）
- Database ID（需要用户提供）

## 技术栈

- **爬虫**: Playwright + Chrome CDP
- **存储**: 本地文件系统（Markdown + JSON）
- **分类**: 关键词匹配 + 可选 LLM
- **问答**: Claude API（基于文章内容的 RAG）
- **Notion**: notion-client (官方 Python SDK)

## 注意事项

1. **文章 ID 生成规则**: `article_YYYYMMDD_序号`
2. **文件名处理**: 特殊字符替换为下划线，避免文件系统冲突
3. **分类冲突**: 如果多个分类得分相同，选择第一个或提示用户
4. **问答上下文**: 每次问答都要传入完整文章内容作为上下文
5. **Notion 限流**: API 有速率限制，批量同步需要控制频率
