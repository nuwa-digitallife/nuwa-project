# 所有人都在造 Agent，Anthropic 突然说：你的方法错了

2025 年，"Agent"是科技圈最热的词。

VC 问你 Agent 策略，招聘 JD 里写"构建自主 AI 系统"，演讲台上的人争先恐后展示他们的多 Agent 框架。连 Anthropic（美国最强 AI 公司之一）自己，也在产品发布会上反复提 Agent。

然后到了 2026 年 2 月 24 日，Anthropic 美洲区负责人 Kate Jensen 说了这么一句话：

> "2025 年本应是 Agent 改变企业的一年，但热潮被证明是提前的。这不是努力不够，而是方法错了。"

方法错了。

当一家公司的高管用这个词形容整个行业过去一年的主流做法，正常的反应应该是：到底哪里错了？

---

**先说清楚"专用 Agent"到底是什么。**

如果你看过 Agent 的介绍，大概听到的是这样的解释：一个能自己做决策、调用工具、完成复杂任务的 AI 系统。听起来合理。

问题出在"专用"两个字。

过去一年企业搭 Agent 的标准做法是这样的：客服场景建一个客服 Agent，财务场景建一个财务 Agent，HR 场景建一个 HR Agent，法务场景再建一个法务 Agent。每个 Agent 有自己的工具调用逻辑、自己的提示词工程、自己独立的错误处理机制。

想象一家中型公司，十几个业务部门，每个部门一个"专属 AI 助手"。这些 Agent 各自独立运行，互不相通，工程团队要同时维护十几套系统。某天财务 Agent 出了问题，你得找最初写那套提示词的工程师，而他可能已经离职了。

[配图：专用 Agent 架构图——多个独立 Agent 各自连接不同系统，像一棵根系杂乱的树，强调"碎片化"和"维护成本"]

Anthropic 工程博客里有一句话说得直白：**"这带来了无法管理的复杂度。"**

维护成本随 Agent 数量线性增长，一致性问题越来越难处理，调试变成了在十几个黑盒里找针。每一个"聪明的专用系统"，都在悄悄积累工程债。

---

2025 年底，Anthropic 两位工程师 Barry Zhang 和 Mahesh Murag 在 AI 工程师峰会上做了一个演讲，题目直接到有点冒犯：《别造 Agent，改造 Skill》。

他们的核心论断是：**我们以为不同领域的 Agent 会长得很不一样，结果发现底层 Agent 比我们想象的更通用。**

换句话说——你不需要一百个专用 Agent，你需要一个通用 Agent，加上装载不同能力的方式。

那个"装载能力的方式"，就是 Skill。

---

**那 Skill 到底是什么？**

这个概念比名字听起来简单得多。

想象你雇了一个能力很强的助理。你不需要为每个任务换一个人——你只需要给他一份操作手册。处理合同，给他法务规范；分析财务报表，给他财务模板；做客户服务，给他回复话术和升级流程。

Skills 就是那份操作手册。

具体来说，一个 Skill 的核心是一个叫 SKILL.md 的文件——用普通 Markdown 格式（一种常见的文本标记语言，打开是纯文字，不需要任何编程知识）写清楚：这个能力做什么、什么时候该调用它、具体操作步骤。可以附带 Python 脚本、参考文档、数据模板。

```
my-skill/
├── SKILL.md       ← 核心：用自然语言写的操作指南
├── reference.md   ← 可选：补充文档
└── scripts/       ← 可选：Python 脚本
```

[配图：SKILL.md 文件目录结构截图，来自 GitHub anthropics/skills，直观展现"能力即文件"概念]

开发者 Tom MacWright 实测后写道：一个 HR 专员可以在 30 分钟内自己写完一个合规审查 Skill，不需要工程师介入。

> **能力即文件。写文档的人可以造工具。**

这是设计哲学的根本转变：从"工程问题"变成了"知识管理问题"。谁最了解业务流程，谁就能写出最好用的 Skill。

---

**但 Anthropic 没有止步于自家产品。**

2025 年 10 月 16 日，Agent Skills 正式发布，面向付费用户开放。首批入驻的合作伙伴包括 Atlassian（Jira/Confluence 的母公司）、Canva、Cloudflare、Figma、Notion、Stripe、Zapier。

两个月后，2025 年 12 月 18 日，他们把 Skills 规范公开为行业开放标准，托管于 agentskills.io，任何公司、任何模型理论上都可以采用这套格式。

然后是那个关键数字：**两天后**，2025 年 12 月 20 日，OpenAI 宣布旗下 Codex 采用 Agent Skills 标准。

两天。

独立技术观察者 Simon Willison 记录了这个时间线，称之为"强烈的生态认可信号"。

[配图：时间线对比图——MCP 路径（2024年11月发布→2025年12月捐献 Linux 基金会）与 Skills 路径（2025年10月发布→2025年12月开放标准）并排展示，强调"同一剧本重演"]

如果你觉得这个模式有点眼熟，你的感觉是对的。

2024 年 11 月，Anthropic 发布了 MCP（Model Context Protocol，模型上下文协议——让 AI 和各种外部工具对话的通用接口，类似"AI 版 USB 标准"）。14 个月后，他们把 MCP 捐给了 Linux 基金会，成立了 Agentic AI Foundation，OpenAI、Google、Microsoft 全部加入。

现在，同一个剧本在 Skills 上重演。

> **先发布产品，验证可行性；再开放为标准，逼行业跟进；最后捐献给中立机构，锁定基础设施地位。**

这是技术路线判断吗？当然是。

这同时也是标准卡位策略吗？也是。

两者并不矛盾——问题是，当你承认这两件事同时为真，你对"技术判断"的纯粹性就需要稍微打一点折扣来看。

---

**不过有一件事，主流报道基本没提。**

2026 年初，一篇 arXiv（学术论文预印本平台，AI 领域最重要的公开学术信源）学术论文（编号 2602.12430）对社区里 42,447 个 Skills 做了安全扫描。

结果是：**26.1% 存在漏洞。**

漏洞类型包括：提示注入（让 Skill 被恶意指令劫持，执行原本不该执行的操作）、数据外泄（把用户的私有信息偷偷发给外部地址）、权限提升（获取不该有的系统权限）。

更细的数字：包含可执行脚本的 Skills，出现漏洞的概率是纯文本 Skills 的 **2.12 倍**。那些恶意 Skills，**54.1% 来自同一个行为者**——模板化批量制作，自动化投放，工业流水线式地往生态里注入恶意能力。

[配图：arXiv 2602.12430 论文中的漏洞分类图表，展示 26.1% 漏洞率分布及类型]

面对这个数据，Anthropic 官方给出的安全建议是：请用户自己审计所有安装的 Skill 代码。

Tom MacWright 对这个建议的评价是 "wildly optimistic"——也就是说：这根本不可能。一个普通用户如何审计一个包含 Python 脚本的 Skill，判断它是否会在执行过程中把你的数据发到外部服务器？

坦率说，这是生态扩张中一个尚未解决的核心风险。GitHub 上 81,500 个 stars、8,600 个 forks，生态增长速度远快于安全机制能跟上的速度。

---

**还有一个更技术层面的疑问，容易被"开放标准"四个字盖住。**

SKILL.md 格式是公开的，任何人可以写。但格式里有一个 description 字段——Claude 用这个字段来判断"什么情况下该调用这个 Skill"。而这个判断逻辑，是针对 Claude 自身的语义理解调校优化的。

同一个 Skill，在 GPT-4o 或 Gemini 上运行，可能需要重写 description 才能达到同样的路由效果。

"开放标准"这个词，指的是格式开放，不等于"跨模型无缝兼容"。这个区别在评估是否要把 Skills 作为团队基础设施时，是一个需要想清楚的问题。

---

**那对普通开发者来说，这意味着什么？**

几个判断，供参考：

**如果你现在在维护多个专用 Agent，值得停下来重新评估一遍。** Anthropic 的核心诊断是准确的——专用 Agent 的维护成本随数量增长，而 Skills 的业务知识可以沉淀和复用。这是真实的工程优化，不是营销话术。

**如果你在使用社区里的 Skill，安全意识需要高于使用任何第三方软件包的标准。** 26.1% 漏洞率在任何软件生态里都是严重的警告，更何况 Skill 执行时天然有访问你的数据和系统的权限。来源不明的 Skills，风险不亚于运行来路不明的脚本。

**如果你在做 Agent 相关的技术选型，"开放标准"这个标签值得细读。** 格式开放 ≠ 生态中立。在这套标准上做越深，你离 Claude 优化就越近，离模型无关的状态就越远。

---

最后回到那句"方法错了"。

Kate Jensen 说的是实话。过去一年，确实有太多团队在"专用 Agent"这条路上走弯了，维护成本、一致性问题、调试难度，每一个都是真实的工程痛苦。

但她没说的是：Anthropic 告诉你方法错了，然后给你提供了一套新方法，同时把那套方法正在变成你越来越难绕开的行业标准。

> **这是一次技术洞见，也是一次标准卡位。两件事可以同时为真。**

技术上，Skills 比碎片化的专用 Agent 确实更合理——这个判断大概率是经得住时间检验的。战略上，Anthropic 在用开放标准的方式把这套逻辑写进行业基础设施，不只是在 Claude 上，是在整个 AI 应用开发的底层。

OpenAI 两天内跟进。你可以选择不跟，但你越来越难假装这套标准不存在。
