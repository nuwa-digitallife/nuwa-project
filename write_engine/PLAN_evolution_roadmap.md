## 一、素材源增强与收敛搜索

### 1.2 收敛搜索（梯度下降式素材采集）

**当前状态**：`deep_research.py` 做固定轮数搜索（中文 3-5 轮 + 英文 3-5 轮），不管够不够。

**目标**：搜索到"新素材被已有素材覆盖"时自动停止——类似梯度下降，到达平坦区就收敛。

**算法**：

```
已有素材集 S = {初始浅层素材}
while True:
    新素材 = search(generate_keywords_from_gaps(S))
    覆盖率 = compute_coverage(新素材, S)  # 新素材中有多少信息已被 S 包含
    if 覆盖率 > 阈值(如 0.8):
        break  # 收敛
    S = S ∪ 新素材
    if len(S) > 上限 or 轮次 > max_rounds:
        break  # 安全阀
```

**覆盖率计算**：用 Claude 判断（prompt: "以下新素材中，哪些信息点在已有素材中已经出现？列出新增和重复的"），返回新增比例。当新增 < 20%，视为收敛。

**实现路径**：
- `deep_research.py` 改为循环调用，每轮基于"缺口"生成新关键词
- 新增 `_compute_coverage()` 函数（调 claude -p with sonnet + low effort，省钱）
- `topic_config.yaml` 新增 `search.convergence_threshold: 0.8` 和 `search.max_rounds: 8`

### 1.3 素材质量自评估

每次采集完成后，自动评估素材丰富度：

| 维度 | 计算方式 | 目标 |
|------|----------|------|
| 信源数 | 去重 URL 数 | ≥ 15 |
| 一手源占比 | (论文+官博+GitHub) / 总信源 | ≥ 30% |
| 观点多样性 | 不同立场数（官方/支持/反对/中立） | ≥ 3 |
| 数据点数 | 可引用的具体数字/日期 | ≥ 10 |
| 时间线完整度 | 覆盖事件起止的时间点数 | 视选题 |

不达标时发出 WARNING，建议补充采集。

## 三、文章迭代求导（Pass 5）

### 当前状态

内容方法论 lines 113-134 已定义 Pass 5 迭代流程和纪律，但引擎代码未实现。

### 设计

在 Pass 4 产出 v1 后，可选触发迭代循环：

```
v1 (Pass 4 终稿)
    ↓
薄弱点分析：逐段标注"证据硬度"（强/中/弱/纯修辞）
    ↓
定向调研：只查弱段的具体问题（不撒网）
    ↓
素材收敛搜索（1.2）→ 补充素材
    ↓
用补充素材替换弱段，写出 v2
    ↓
v2 跑 Pass 2 (事实) + Pass 3 (Review) + Pass 3.5 (协商) → 独立完整 review
    ↓
v1 vs v2 对比：字数、信源数、证据硬度、人设纯度
    ↓
如果 v2 显著优于 v1 → 再来一轮出 v3
    ↓
收敛判断：vN vs vN-1 差异主要在用词/标点 → 停止
```

**触发条件**（自动判断）：
- 争议性选题 → 自动触发
- 调查型文章 → 自动触发
- 常规信息型 → 跳过
- 用户手动 `--iterate` 强制触发

**实现**：

```python
# engine.py 新增
def run_iteration_loop(ctx, topic_dir, persona, series, model, max_iterations=2):
    """Pass 5: 迭代求导。v1 已存在，产出 v2, v3..."""
    for i in range(2, max_iterations + 2):
        # 1. 薄弱点分析
        weakness_report = analyze_weakness(current_article)
        # 2. 如果没有弱段，收敛
        if weakness_report.all_strong():
            break
        # 3. 定向调研（收敛搜索）
        supplementary = targeted_research(weakness_report.weak_points)
        # 4. 替换弱段写 vN
        v_new = rewrite_weak_sections(current_article, supplementary)
        # 5. 独立 Pass 2 + 3 + 3.5
        v_new = run_pass2_to_3b(v_new)
        # 6. 对比 vN vs vN-1
        comparison = compare_versions(current_article, v_new)
        # 7. 保存
        save(f"article_v{i}.md", v_new)
        save(f"iteration_comparison_v{i}.md", comparison)
        current_article = v_new
```

**新增文件**：
- `engine.py` 新增 `run_iteration_loop()` + `analyze_weakness()` + `compare_versions()`
- `prompts/pass5_weakness.md` — 薄弱点分析 prompt
- `prompts/pass5_rewrite.md` — 定向重写 prompt
- `prompts/pass5_compare.md` — 版本对比 prompt
- CLI: `--iterate` 标志 + `--max-iterations N`

**输出**：
- `article_v2.md`, `article_v3.md` — 各版本
- `iteration_comparison.md` — 汇总对比表

## 四、发布自动化

### 4.1 浏览器自动化能力提升

**当前痛点**（MEMORY.md 记录）：
- Playwright 持久化 context 冲突
- 系统剪贴板配合不稳定
- 脚本结束浏览器就关

**方案：分层工具栈**

| 层 | 工具 | 用途 |
|---|------|------|
| API 层 | JS 脚本注入 (`inject_js.py`) | 能用 API 的都用 API |
| 浏览器层 | Playwright (headless=False, 持久化 profile) | 需要完整页面交互的 |
| 系统层 | cliclick + AppleScript | 系统级操作（剪贴板、窗口） |

**关键改进**：
1. **Browser Pool**：启动一个长驻 Playwright 浏览器进程（不随脚本结束关闭），通过 WebSocket 连接
2. **Chrome CDP 直连**：用 `chrome --remote-debugging-port=9222`，Playwright 连接已有 Chrome（用户已登录微信后台）
3. **剪贴板隔离**：操作前保存、操作后恢复系统剪贴板

**实现路径**：
- `wechat/tools/publish/browser_pool.py` — 浏览器池管理
- 改造现有 `inject_js.py` / `mdnice_render.py` 使用浏览器池
- `publish/one_click_publish.py` — 串联 4.1-4.6 的端到端发布

### 4.2 Mac Mini 并发与多端同步

**场景**：Mac Mini 跑后台任务（素材采集、写作引擎），MacBook 做开发。

**同步方案**：

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **Git** | 自然版本控制，离线可用 | 手动 push/pull，冲突处理 | ✅ 主方案 |
| **syncthing** | 实时双向同步 | 可能冲突，不适合代码 | ❌ |
| **rsync + cron** | 简单可靠 | 单向，无冲突检测 | 🟡 备选 |

**推荐方案：Git + 自动同步脚本**

```bash
# Mac Mini: 每次引擎跑完自动 commit + push
# scripts/sync_after_run.sh
git add wechat/公众号选题/
git commit -m "auto: engine output $(date +%Y-%m-%d_%H:%M)"
git push

# MacBook: 开发前 pull
git pull --rebase
```

**环境配置**：
- `scripts/setup_new_machine.sh` — 一键配置新机器（brew 依赖、venv、Chrome profile、launchd jobs）
- `.env.example` + `scripts/setup_env.sh` — API keys 安全管理

## 七、女娲主项目关联

以下能力从公众号自动化中诞生，但本质上属于女娲计划的通用基建：

| 能力 | 公众号场景 | 女娲通用场景 | 对应女娲概念 |
|------|-----------|-------------|-------------|
| **插件化信息源** (1.1) | 公众号/Twitter/RSS | 任何需要信息采集的 Agent | Agent 的"感知系统" |
| **收敛搜索** (1.2) | 素材采集达到饱和 | Agent 探索环境后判断何时停止 | 压缩→展开→再压缩循环的"收敛判断" |
| **异步人机协作** (1.4) | 晨间任务清单 | Agent 提问不阻塞等待 | 骑手与马的异步接口 |
| **多机协作** (4.2) | Mac Mini 跑引擎 | 多 Agent 分布式协作 | 拉法尔自我繁殖的基础设施 |
| **反馈→经验→策略调整** (5.2) | 阅读数据优化写作 | Agent 的通用强化学习循环 | 元规则："根据反馈调整行为" |
| **迭代求导** (三) | 文章版本收敛 | Agent 的自我优化循环 | "压缩即智能"的实践 |

**建议**：在 `docs/zh/nuwa-plan.md` 中新增一节"活体实验：公众号自动化"，记录这些从实践中涌现的通用模式。不是提前设计的，是做公众号过程中自然发现的——这本身就是"框架涌现"的例证。

## 九、验证方式

| 任务 | 验证方法 |
|------|---------|
| E2E 测试 | 用蒸馏门选题跑引擎 Pass 1-4，与人工版本对比 |

| 发布脚本 | 从 markdown 到微信草稿的完整跑通 |
| 反馈采集 | 手动发一篇后 24h 采集数据，与后台手动查看对比 |
| 多端同步 | Mac Mini 引擎产出 → MacBook git pull 看到 |
