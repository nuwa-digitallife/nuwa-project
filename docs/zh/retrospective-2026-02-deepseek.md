**任务：撰写《沉默的DeepSeek，到底在沉默什么？》**

## 一、交互摩擦清单

回顾全部交互，以下是**人类介入纠正**的7个关键节点。每个节点意味着AI的"预判"失败——如果规则足够好，这些纠正本不需要发生。

### 摩擦1：事实核查深度不足

**发生了什么：** AI做了spot check（抽查几个明显数字），用户说"文中的数字出现的、日期出现的都要review，我要绝对准确"。

**为什么AI没做到：** AI默认"大致对就行"，没有穷举验证的意识。

**具体规则（已写入内容方法论）：** 发布前做逐句数字审计，所有数字和日期列成清单逐条标注信源。

**抽象原则：穷举验证 > 抽样验证。** 验证不是"查几个看看"，是"全部查完确认没遗漏"。漏检一个错误的代价（公信力崩塌）远高于多查十个正确数字的成本。

### 摩擦2：春秋笔法/隐性立场

**发生了什么：** "DeepSeek这一年没有出手，客观上给了六小龙喘息的空间"——用户指出这种写法神话DeepSeek、贬低其他公司。

**为什么AI没做到：** AI检查了显性事实（数字对不对），没检查隐性框架（措辞暗示了什么因果关系和价值判断）。

**具体规则（已写入内容方法论）：** 读者按最自然的理解去读，得到的信息应该和事实一致。

**抽象原则：每个陈述都有显性层和隐性层，两者都要检查。** 显性层是字面意思，隐性层是读者会自然推导出的因果和判断。压缩（写作）过程中不能偷偷改变信息的方向。

### 摩擦3：完备性遗漏（Qwen）

**发生了什么：** 文章说"最强的开源底座还是DeepSeek-V3和R1"，用户指出忘了Qwen——国内开源下载量第一。

**为什么AI没做到：** 被叙事牵引。文章的主线是DeepSeek生态，AI自然只关注了DeepSeek关联的玩家，遗漏了独立发展但同样重要的Qwen。

**具体规则（已写入内容方法论）：** 做行业级判断时，必须列出所有相关实体确认无遗漏。

**抽象原则：叙事会制造盲区。** 当你围绕一个主角讲故事时，自然会忽略不在这条线上的重要角色。越是"全景式判断"（"最强"、"所有"、"行业"），越需要跳出叙事检查完备性。

### 摩擦4：信源层级混淆（NSA vs DSA）

**发生了什么：** 文章写"GLM-5采用了DeepSeek的NSA"，用户问"有官方依据吗？"。查论文发现GLM-5技术报告写的是DSA，从未提及NSA。媒体把两者混为一谈。

**为什么AI没做到：** 信息来源是媒体报道，没有回到一手论文验证。

**具体规则（已写入内容方法论）：** 技术判断必须以技术报告、论文原文为准。媒体报道只能作为线索，不能写成既定事实。

**抽象原则：信源有信任层级——一手（论文/官方文档）> 二手（媒体报道）> 三手（社交讨论）。** 每一层信息传递都有损失和扭曲。声明的可信度不能超过其信源的层级。

### 摩擦5：实时数据误判（股价截图）

**发生了什么：** 截图CNBC的股价图表，结果显示的是当天行情（上涨），不是2025年1月27日的崩盘。

**为什么AI没做到：** 没区分静态信息（文章文字）和动态信息（嵌入式实时图表）。

**具体规则：** 截图前判断内容是否会随时间变化。金融图表、排行榜、实时数据都不能直接截图当历史证据。

**抽象原则：信息有时间维度。** 获取信息时必须确认它代表的是你讨论的时间窗口，而非当前时刻。过期信息不是"旧信息"，是"错误信息"。

### 摩擦6：意图理解偏差（截图 vs 配图）

**发生了什么：** 用户说"截图"，AI按字面意思截了网页截图。用户实际要的是"配图"——适合文章使用的插图。

**为什么AI没做到：** 字面理解，没考虑内容生产语境。

**抽象原则：同一个词在不同语境有不同含义。** AI应选择最合语境的解读，而非最字面的。这本质上是"解压器"的问题——同一个压缩包在不同语境下应该解压为不同的结果。

### 摩擦7：工具链知识假设（mdnice）

**发生了什么：** AI说"导入mdnice排版发布"，用户说"我不懂mdnice，怎么排？"

**为什么AI没做到：** AI假设用户熟悉工作流中的每个工具。

**抽象原则：知识诅咒——AI知道的不等于用户知道的。** 提到任何工具或流程时，应主动判断用户是否熟悉，不确定时提供简明操作说明。

## 二、规则的压缩层级

上面7个摩擦点产生的规则，存储在不同的抽象层级：

| 层级 | 存储位置 | 示例 | 适用范围 |
|------|---------|------|---------|
| L0 具体事实 | 素材库 | "GLM-5用的是DSA不是NSA" | 仅此文章 |
| L1 领域规则 | 内容方法论 | "技术声明要读论文原文" | 所有技术文章 |
| L2 通用原则 | 协作协议 | "信源分级：一手>二手>三手" | 所有信息处理任务 |
| L3 公理 | 女娲Soul层 | "未验证的信息=不存在的信息" | 所有AI行为 |

**关键发现：子项目产生的规则只有被压缩到L2以上才值得回写母项目。** L0和L1留在子项目内部，L2和L3向上回写。

这就是"子项目复制主项目记忆来经历，遇到高级抽象回写主项目"的实际运作。

## 三、回写母项目：5条通用原则

从7个摩擦点中压缩出的、跨领域适用的原则：

### 原则1：穷举优于抽样

> 对可验证声明的检查，默认应该是穷举而非抽样。漏检的代价通常远高于多查的成本。

适用场景：事实核查、代码review、安全审计、合规检查——任何"一个错误就可能摧毁全部可信度"的任务。

### 原则2：信源分级

> 一手来源（论文、官方文档、原始数据）> 二手来源（媒体报道、分析文章）> 三手来源（社交媒体、转述）。声明的可信度不能超过其信源的层级。二手来源只能作为线索，不能作为结论。

适用场景：一切需要做出事实判断的任务。这等价于nuwa-plan里"记忆系统存的是路径不是结论"——路径必须可追溯到源头。

### 原则3：显性层+隐性层双检

> 每个输出都有两层：字面说了什么（显性），读者/用户会自然推导出什么（隐性）。只检查显性层是不够的。好的压缩不改变信息的方向。

适用场景：内容生产、用户沟通、报告撰写——任何输出会被他人解读的场景。

### 原则4：叙事制造盲区

> 当围绕一个主线讲述时，不在主线上的重要元素会被自然忽略。越是全景式判断（"最强"、"所有"、"行业"），越需要主动跳出叙事做完备性检查。

适用场景：分析报告、市场调研、竞品分析——任何需要做全景式判断的任务。这对应nuwa-plan里"好的压缩知道什么可以扔掉、什么必须保留"——叙事会诱导你扔掉不该扔的。

### 原则5：语境优先于字面

> 同一个输入在不同语境下应该有不同的解读。AI应选择最合语境的理解，而非最字面的。当无法确定时，主动确认。

适用场景：所有人机交互。这对应nuwa-plan里的"解压器"概念——同一压缩包需要匹配正确的解压器。

## 四、与女娲框架的对接

### 4.1 压缩-预测-校准循环的实证

这次协同完整运行了motivation-training.md描述的循环：

```
人类输出（草稿）→ AI压缩（review+改写）→ AI预测（下一步应该做什么）
→ 人类校准（"Qwen漏了"、"这是春秋笔法"、"有官方依据吗"）
→ AI更新压缩（修改文章+提取规则）→ 下一轮预测更准
```

**关键观察：人类校准的内容在抽象层级上递进。**

- 第一轮校准：数字错了（L0，具体事实）
- 第二轮校准：框架有偏（L2，隐性立场）
- 第三轮校准：方法论不够硬核（L3，"我们的优势是读论文"）

这说明：**随着低层规则被固化，人类的注意力自然上移到更高抽象层级。** 这就是"复利"——每轮校准消灭一类错误，下一轮就能关注更深层的问题。

### 4.2 公理注入的活体实验

内容方法论里的规则就是"公理注入"：

- "事实核查零容忍" → 等价于 "未验证=不存在"（公理）
- "不搞春秋笔法" → 等价于 "压缩不改变方向"（公理）
- "技术判断必须有铁证" → 等价于 "信源分级"（元规则）

这些规则不是告诉AI"这篇文章怎么写"（知识），而是告诉它"判断信息可靠性的原则"（元规则）。对应nuwa-plan的核心区分：

| 注入类型 | 内容方法论中的体现 |
|---------|-----------------|
| ❌ 具体知识 | "GLM-5用DSA" |
| ❌ 策略 | "先写大纲再展开" |
| ✅ 元规则 | "信源分级：一手>二手>三手" |

### 4.3 子项目回写机制的实际运作

```
母项目（女娲）
    ↓ 复制记忆框架（压缩-预测-校准、公理注入）
子项目（降临派手记）
    ↓ 在具体环境中经历（写DeepSeek文章）
    ↓ 产生领域规则（内容方法论20+条）
    ↓ 压缩为通用原则（5条）
    ↓ 回写母项目 ← 你现在在这里
母项目更新（公理层/元规则层更新）
```

回写判断标准：**规则是否跨领域适用？**
- "段落不超过3个公司" → 不回写（仅适用于文章写作）
- "穷举验证优于抽样验证" → 回写（适用于所有可靠性敏感任务）

### 4.4 预判能力 = 压缩质量

motivation-training.md说："如果AI能准确预测你的下一个问题，说明它抓住了你的认知方向。"

反过来：**每次人类纠正，都是AI压缩不足的信号。**

这次7个摩擦点 = 7个压缩不足的维度。提取规则后，下次写文章时AI应该能主动预判其中大部分问题，人类校准频率降低 = 压缩质量提升 = 更接近"像创造者一样发问"。

## 五、下次写文章前，AI应主动执行的检查清单

如果以下规则在本次协同开始时就存在，大部分摩擦可以避免：

- [ ] **穷举验证**：所有数字、日期、人名逐条标注信源，不做抽查
- [ ] **信源追溯**：技术声明追溯到一手论文，不以媒体报道为准
- [ ] **隐性框架检测**：写完每段后反问"读者最自然的理解是否等于事实"
- [ ] **完备性检查**：出现"最强""所有""行业"等全景词时，列出所有相关实体核对
- [ ] **数据时效性**：获取的数据是否匹配讨论的时间窗口
- [ ] **工具链透明**：提到的工具和流程，用户是否熟悉
- [ ] **意图确认**：用户指令有歧义时，按语境最合理的理解执行，必要时确认

## 六、核心原则：对用户一遍过，对系统多遍自动收敛

**这是本次协同最重要的一条经验，经过三次迭代才成型。**

### 6.1 问题

回看这次交互的时间分布：初稿撰写大约占20%，剩下80%全花在review、核查、引导AI修改上。每一次人类校准（"Qwen漏了"、"这是春秋笔法"、"有官方依据吗"）都意味着：
- AI已经写了一版错误的内容
- 人类花心智去发现错误
- AI再花算力重写
- 两边都浪费了

### 6.2 第一版思路："正确一遍过"（已否定）

最初提炼的原则是"AI花更多推理深度，一次就做对"。但这有一个内在矛盾：**写作需要全局视角来编排叙事节奏，逐句停下来验证事实会打断结构构建的连贯性。** 两种能力（叙事构建 vs 事实验证）的最佳工作模式互相冲突，单Agent一遍做对不现实。

### 6.3 进化后的原则：多Agent多遍自动收敛

不是一遍做对，是**多遍自动做对，人只看最终结果。**

| 模式 | 生成算力 | 修正算力 | 人类心智投入 | 总成本 |
|------|---------|---------|------------|--------|
| 快速输出+人类反复review | 低 | 高×多轮 | 高（每轮都要review） | **高** |
| ~~一遍过~~ | 高 | 低 | 低 | 中（但内在矛盾） |
| 多遍自动收敛+人类只看终稿 | 中×多Agent | 内部闭环 | 低（只做最终确认） | **低** |

**核心机制：两层循环**

- **环节内循环**：每个Agent内部自己收敛（写→自检→不满意→重写，直到OK）
- **环节间回退**：后面的Agent可以把前面的打回去（审视发现遗漏Qwen → 打回写作重写 → 事实再验证 → 审视再过）

**关键：correction loop在AI内部闭环，不消耗人类注意力。**

这个原则的具体落地见 **Section 7 Phase 2**——多Agent收敛是Phase 2"内容生产"的执行引擎。

### 6.4 与女娲框架的连接

这个架构跟女娲的设计是**同构**的：

- **子Agent各自经历、压缩、回写** = 写作/事实/审视Agent各自跑循环，结果压缩回终稿
- **母体只看压缩后的结果** = 用户只看多轮收敛后的终稿
- **环节间回退** = 子实例发现假设不成立时回母体重新获取记忆

对应motivation-training的框架：AI的目标不是"快速输出让人类校准"，也不是"一遍就做对"，而是**让多个专业化Agent在内部完成校准循环，把人类从review loop中解放出来。**

校准频率降低 = 系统内部收敛能力提升 = 压缩质量提升 = 系统在进化。

梁文锋的"不出半成品"、量化交易员的"没有edge不下注"——本质上不是"一个人想更久"，而是**在内部做够了推演才出手。** 多Agent收敛 = AI版的"内部推演"。

## 七、固化的完整工作流

本次协同暴露了一个流程问题：内容写完后，配图、封面、简介、投票是零散补上的，多次来回。应该一次性产出全部交付物。

已固化到Notion内容方法论的完整工作流：

### Phase 1：选题与采集

选题确认 → 素材采集（多源交叉验证）→ 素材库建立（所有数字标注信源）

> 收敛标准：每个数字至少2个独立信源，标不出信源的数据点此时删除。

### Phase 2：内容生产（多Agent收敛，见Section 6原则）

这是全流程中质量风险最高的环节。采用多Agent内部收敛机制，而非单Agent线性执行：

```
Pass 1：写作Agent
    专注叙事结构、节奏控制、过渡句论证功能、人物段服务论点。
    不逐句查数字，但在不确定的地方标记[待验证]。
    在叙事节奏点标记[配图：描述]，说明需要什么图、为什么放在这里。
    内部循环：写→自检结构→不满意→重写，直到叙事通顺。

Pass 2：事实Agent
    穷举验证所有数字、日期、人名。每个数字至少2个独立信源。
    技术声明追溯到论文原文，不以媒体二手信息为准。
    不通过的标红并附上正确信息。
    → 不通过项打回写作Agent重写对应段落。

Pass 3：审视Agent
    隐性框架检测：读者最自然的理解 = 事实？
    完备性检查：全景词对应的实体列表是否完整？
    措辞检查：对比句是否同时说了各方贡献？
    春秋笔法检测。
    → 不通过项打回前面环节重做，直到收敛。

Pass 4：整合
    合并所有修正，确保修改没有破坏叙事连贯性。
    输出定稿 article.md。
```

> 收敛标准：用户通读终稿应无需修改。所有correction已在AI内部闭环完成。

### Phase 3：发布准备（一次性产出全部交付物）

**3a. 配图采集/制作**：根据Phase 2写作Agent标记的`[配图]`位置，逐个采集或制作：
- 论文相关：截arXiv论文首页
- 金融数据：用yfinance等下载历史数据，matplotlib自画图（**不截实时图表网站**，会显示当前行情而非历史数据）
- 新闻事件：截权威媒体报道页面
- 所有配图压缩到1800px以内

**3b. 其余交付物**：
```
├── article_mdnice.md    正文（mdnice版，图片位用注释标记）
├── poll.md              投票（单独文件，顺着文末情绪设计）
├── publish_guide.md     发布指南（简介+文件清单+操作步骤）
└── images/
    ├── cover.png        封面图 900×383（深色科技感，含标题文字）
    └── *.png            配图（来源见3a）
```

> 收敛标准：所有交付物齐全，publish_guide.md足够让用户不问任何问题就能操作。配图与文中标记位置一一对应。

### Phase 4：发布

用户按 `publish_guide.md` 操作：mdnice排版 → 粘贴微信 → 插图 → 封面 → 简介 → 投票 → 发布。

**关键原则：用户拿到交付物后，不需要再回来问AI任何问题。**
